## 1. 신설법인 개요

### 대상 법인
| 법인명 | 구매조직 코드 | 비고 |
|--------|-------------|------|
| KT Netcore | 1400 | 공사 총액계약 포함 |
| KT P&M | 1600 | 공사 관리자 명칭 사용 |

### 신설법인 특징
- 자재코드 **미관리** (자재구매 없음)
- KT 프로그램에서 **업무대행 ID**로 진행
- 벤더 코드: **4~, 5~** 로 시작하는 코드 사용
- 코스트센터 기준으로 **예산 체크** 적용
- MM 기준 커스텀 테이블: `ZTSBMMT*` 로 시작

## 2. DIP 관련 문의

### Q. DIP 요청은 어디서 생성하나요?
- **생성**: `ZSBMMTE019061_C/P`
- **변경/조회**: `ZSBMMTE019062_C/P`

### Q. DIP 와 ATACAMA 설계 (WBS) 는 몇 대 몇으로 매핑해야 하나요?
- 반드시 **1:1 매핑** 원칙 준수
- 1:2 매핑 시 **변경계약에서 문제 발생**

### ⚠️ 문제 사례: DIP 1 개 - WBS 2 개 매핑 오류
**증상:**
1. ATACAMA 에서 `-16` 프로젝트가 DIP 요청번호 `2026003317` 로 연동된 것으로 표시됨
2. DIP 요청관리에서 `2026003317` 검색 시 `-17` 프로젝트로 검색됨
3. `-16` 프로젝트는 SAP 에서 검색 불가

**원인:**  
DIP 하나에 ATACAMA 설계가 2 개 연결됨 → DIP : 설계 = 1:2 구조로 잘못 생성

**조치 방법:**
1. 해당 PR 반려 처리
2. DIP(`2026003317`) 삭제
3. DIP 와 설계를 각각 1:1 로 신규 생성
4. 각각 1:1 매핑 확인 후 계약 재진행

### Q. DIP 관련 테이블은 어디에 있나요?
| 테이블 | 설명 |
|--------|------|
| ZTSBMMT0092 | DIP 헤더 |
| ZTSBMMT0093 | DIP 품목 |
| ZTSBMMT0092H | DIP 헤더 이력 |
| ZTSBMMT0093H | DIP 품목 이력 |
| ZTSBMMT0220 | BPM 상태관리 (DIP 유형코드: F_136) |

## 3. 계약 관련 문의

### Q. 총액계약과 단가계약의 차이가 뭔가요?
| 항목 | 총액계약 | 단가계약 |
|------|---------|---------|
| 예산 | 계약요청 시 **필수** | 불필요 |
| 변경 | SAP + SRM 모두 가능 | **SRM 에서만** 가능 |
| 예산차감 시점 | PR(구매요청) 시점 | PO 시점 |

### Q. 단가계약 프로세스는?
```
PR → Contract → PO → GR → I/V
```

### Q. 총액계약 전체 프로세스는?
```
DIP 요청 → BPM 결재 → PR 생성 → SRM 계약 생성 → ERP PO → (선금) → SES → GR → I/V → 지출결의 → 결재 → 지급
```

### Q. BPM 문서 구조가 어떻게 되나요?
- **BPM-105**: 껍데기 생성 (계약의 기본 틀)
- **BPM-101**: 상세내역 입력
- 단가계약에서 생성되는 계약서 (5 번대) 는 PO 생성 전에 발생
- `ME33K` = 계약 조회 (약속서), PO = 실제 주문서

### Q. 수의계약 vs 도급계약 차이는?
- **수의계약**: 경쟁입찰 없이 특정 업체 지정 계약 (긴급 조달 등 특수 상황)
- **도급계약**: DIP 요청 시 하도급 계약 유형 선택 가능

### Q. CMS 가계약번호는 무엇인가요?
- 정식 계약 체결 전, CMS 를 통해 선금/보증금 처리 시 부여하는 내부 관리번호
- **공사에는 없고, 용역에만 존재**
- 이후 정식 계약번호와 매핑하여 관리

## 4. 예산 / 계정 관련 문의

### Q. 신설법인 예산차감 기준은?
- **코스트센터 기준** 예산 체크 적용
- 예산차감 시점: **PR(구매요청) 시점**

### Q. 계정지정범주 (Account Assignment Category) 종류는?
| 코드 | 귀속 | 사용 예시 |
|------|------|---------|
| **P** | 프로젝트 WBS | 대규모 프로젝트 공사, 장기 용역 |
| **K** | 코스트센터 | 부서 예산, 일반 유지보수 |

### Q. ATACAMA 설계발주 시 예산은?
- WBS 가 키 값으로 사용됨 → **무조건 프로젝트 예산**으로 진행
- **Netcore 에만 해당** 사항
- 설계발주 시 해당 WBS 에 예산 확보 필요

## 5. SES / 입고 관련 문의

### Q. 기성/준공 처리 흐름은?
1. SRM 에서 협력사가 기성·준공금 신청
2. ERP SES 생성
3. 계약담당자 SES 승인 (`ZSBMMTE0220_C/P`)
4. ERP 입고전표 (GR) 생성

### Q. SES 처리 T-Code 는?
- `ML81N`: 서비스 엔트리 시트 생성 및 입고
- `MIGO`: 물자입고 (이동유형 101: PO 기준 입고)

### Q. SES 관련 테이블은?
| 테이블 | 설명 |
|--------|------|
| ZTSBMMT0510 | SES 생성 SRM I/F 수신 데이터 이력 (헤더) |
| ZTSBMMT0511 | SES 생성 SRM I/F 수신 데이터 이력 (아이템) |
| ESSR | 서비스 입력 시 헤더데이터 |

## 6. 송장 / 세금계산서 관련 문의

### Q. 역발행 vs 정발행 차이는?
| 구분 | 역발행 | 정발행 |
|------|--------|--------|
| 발행 주체 | KT(발주처) | 협력사 (공급업체) |
| 처리 방식 | KT 가 세금계산서 발행 요청 → 협력사 승인 | 협력사 직접 발행 → KT 승인 |

### Q. 세금계산서 조회/맵핑 T-Code 는?
- `ZEVRR0030`: 세금계산서 조회, 수기맵핑, 맵핑취소

### Q. 임시송장 생성 T-Code 는?
- `MIR7`: 임시 송장 생성
- `MIR4`: 송장 조회 (임시 포함 모든 송장)
- `MIRO`: 전기된 송장 생성

### Q. 지출결의 처리 순서는?
1. 지출결의 생성: `ZEVRR0050_C/P` (입고 승인된 건만 표시)
2. 지출결의 결재 요청: `ZSBMMTW0030_C/P` (세금계산서 맵핑된 임시송장만 BPM 결재 가능)
3. 회계팀 승인 → 임시송장 전기 (확정)

### Q. 관련 테이블은?
| 테이블 | 설명 |
|--------|------|
| ZTEVR0030 | 전자세금계산서 AR Header |
| ZTEVR0050 | 전자세금계산서 AP 상태관리 Log |
| ZTEVR0060 | 전자세금계산서 AP 상태관리 Header |
| ZTEVR0070 | 전자세금계산서 AP 데이터 Header |
| ZTSBMMT0230 | 협력사 임시송장 결재 상세 내역 |
| RBKP | 송장 헤더 |
| RSEG | 송장 품목 |

## 7. 선금 (선급금) 관련 문의

### Q. 선금 지급 조건은?
- 영세 업체의 유동성·현금 부족 문제 발생 시
- **최초 1 회만** 지급 가능

### Q. 선금 처리 T-Code 는?
- 승인/반려: `ZSBMMTE0220_C/P` (계약담당자)
- 선금차감: `ZSBMMTE0600_C/P` (지출결의 결재 **전** 반드시 처리)

### Q. 선금 처리 예시 (계약금액 100 원)
| 구분 | 금액 | 비고 |
|------|------|------|
| 선금 지급 | 20 원 | 최초 1 회 |
| 1 차 기성 송장 | 50 원 | 선금차감 10 원 → 실지급 **40 원** |
| 2 차 기성 송장 | 50 원 | 선금차감 10 원 → 실지급 **40 원** |
| **합계** | **100 원** | 선금 (20) + 1 회차 (40) + 2 회차 (40) |

> ⚠️ 송장 없이 지출결의만 단독 진행 시 반제 처리에 문제 발생!

## 8. 지체상금 관련 문의

### Q. 지체상금 발생 조건은?
- PO 납품일 초과 입고 시 업체 지체로 간주
- 지체송장 자동 생성 → 지체상금 차감 후 지급

### Q. 지체상금 처리 방법은?
- **방법 1 (차감)**: 지체송장 + 임시송장 묶어서 결재 요청 → 반제 처리
- **방법 2 (예외처리)**: 예외처리로 pass 가능 (사유 입력 필수)

### Q. 지체상금 관련 T-Code 는?
- 생성 및 예외처리: `ZSBMMTE0090_C/P`
- 삭제: `ZSBMMTE0140_C/P`

## 9. 벤더 / 협력사 관련 문의

### Q. 신설법인 벤더 코드 형태는?
- `4~`, `5~` 로 시작하는 코드 사용
- `LFA1` 테이블에서 사업자등록번호로 조회 가능

### Q. SRM 을 통한 벤더 처리 흐름은?
1. 협력사가 SRM 에 회원 가입
2. SAP ERP 와 연동 → BP(Business Partner) 자동 생성
3. 구매조직 확장 가능 → 구매 진행 가능

> ⚠️ ERP FI 모듈에서 직접 생성 시: 구매조직 확장 불가 → 구매 불가능

### Q. 협력사 관련 테이블은?

|--------|------|
| ZTSBMMT0040 | SRM 협력사 정보 |
| ZTSBMMT0560 | ERP 협력사 생성 및 변경 Log (ERP → SRM) |
| ZTSBMMT0630 | 사업자 변경 CBO 테이블 |

## 10. ATACAMA 설계 관련 문의

### Q. 신설법인 ATACAMA 설계 원칙은?
1. 설계가 WBS 를 물고 진행하기 때문에 **(1:1) 원칙**
2. WBS 1 건에 품목 여러 개는 가능

### Q. ATACAMA 설계 관련 테이블은?
| 테이블 | 설명 |
|--------|------|
| ZTSBMMT0090 | DSS 설계 구매요청 DATA 관리 HEADER (최초 계약 데이터) |
| ZTSBMMT0091 | DIP Interface Item |
| ZTSBMMT0090I | DIP Head Interface 정보 (ATACAMA 이력) |
| ZTSBMMT0091I | DIP Item Interface Log |
| ZTSBMMT0090H | DIP Interface Head History (변경계약 전 이전 차수 설계) |
| ZTSBMMT0091H | DIP Interface Item History |

### Q. '감리자'와 '공사 관리자' 차이는?
- P&M 에서는 **'감리자'** → **'공사 관리자'** 로 명칭 변경
- 기존 '감리자' 명칭은 법적으로 문제가 될 수 있어 변경된 것
